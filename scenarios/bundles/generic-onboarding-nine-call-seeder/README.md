# generic-onboarding-nine-call-seeder

Generated by PocketHive wizard from: Create a generic onboarding data seeder HTTP sequence bundle that proves a tenant batch flow and per-user onboarding flow with conditional validation polling. The proof should execute 9 outbound HTTP calls in total for one user: open batch, create profile/start validation, poll validation pending, poll validation approved, create account, initial funding/setup, provision resource, enable optional capability, and close batch. Activation is deferred as a follow-on process. This is a data seeding capability, not a performance test.

## Polling Proof

The sequence has 8 configured business steps. The `poll-validation-result`
step retries while JSON `validationStatus` is `PENDING`, so the proof run
executes 9 outbound HTTP calls: the first poll returns `200 PENDING`, then the
retry returns `200 APPROVED`. This demonstrates bounded conditional polling by
response body. Activation remains deferred for a follow-on process.

## Tenant Scale Proof

`variables.yaml` defines ten tenant profiles. Each profile sets `userCount:
10000`, and the generator scheduler uses `maxMessages: "{{ vars.userCount }}"`
so one swarm profile run emits exactly the configured number of user journeys
for that tenant.

Successful journeys are forwarded by the postprocessor into the tenant-specific
Redis list from `vars.outputList`. Each list entry contains the final HTTP
sequence result with approved validation, active account status, active resource
status, and active card status extracted from the onboarding flow.

The current sequence keeps the batch open and close calls inside each generated
journey so the end-to-end row is self-contained. A separate tenant-control
orchestration can move those calls outside the per-user loop if the production
seeder needs exactly one open/close batch per tenant.
